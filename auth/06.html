How Digital Signatures Work

A digital signature is a cryptographic mechanism that provides authenticity (proves the sender's identity), integrity (ensures the data hasn't been tampered with), and non-repudiation (the sender can't deny signing it). It uses asymmetric encryption: The signer uses their private key to create the signature, while anyone with the signer's public key can verify it. This is efficient because only the private key holder signs, but verification is public and fast.

Digital signatures don't encrypt the data—they sign it, often on a hash of the data to handle large payloads. Common algorithms include RSA, ECDSA (Elliptic Curve Digital Signature Algorithm), and EdDSA.

Step-by-Step: How Digital Signatures Work

  1. Hash the Message:

    • Compute a fixed-size digest (hash) of the original data using a cryptographic hash function (e.g., SHA-256). This condenses the message into a short, unique fingerprint.
    • Why? Signing the full message is inefficient; the hash captures any change (avalanche effect).

  2. Sign the Hash:

    • The signer applies their private key to the hash using the algorithm (e.g., RSA-PSS padding).
    • Output: A signature (a number or byte array, e.g., 256-512 bits).

  3. Attach and Transmit:

    • Bundle the original data + signature (and optionally the public key or its ID).
    • Send over an insecure channel.

  4. Verify the Signature:

    • Recipient recomputes the hash of the received data.
    • Use the signer's public key to "decrypt" (verify) the signature, extracting the embedded hash.
    • If hashes match and the math checks out, it's valid. Mismatch? Tampered or wrong key.

If the signature fails, the message is rejected—preventing man-in-the-middle attacks.

Example: Digital Signatures in JWT (JSON Web Signature - JWS)

JWTs often use digital signatures in their JWS format to secure tokens (e.g., for auth in APIs). A signed JWT looks like: header.payload.signature (base64-encoded parts separated by dots).

  • Header: JSON with algo (e.g., {"alg":"RS256","typ":"JWT"}) → Base64.
  • Payload: Claims (e.g., {"sub":"user123","exp":1234567890}) → Base64.
  • Signing Input: Base64(header) + "." + Base64(payload).
  • Signature Creation:
    1. Hash the signing input (SHA-256).
    2. Sign the hash with issuer's private key (RS256 = RSA + SHA-256).
    3. Base64-encode the signature.
  • Verification (Offline-Capable):
    1. Recompute hash of received header.payload.
    2. Use issuer's public key to verify the signature against the hash.
    3. Check claims (e.g., expiry) if valid.

Example JWT (Simplified):

Header: {"alg": "RS256"}
Payload: {"sub": "alice", "iat": 1516239022}
Signing Input: eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJhbGljZSIsImlhdCI6MTUxNjIzOTAyMn0
Signature: SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c  (signed hash)
Full Token: eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJhbGljZSIsImlhdCI6MTUxNjIzOTAyMn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

In .NET: Use System.IdentityModel.Tokens.Jwt to sign/verify: new JwtSecurityTokenHandler().WriteToken(token) with RSA keys.

Comparison of Signature Algorithms in JWT

Algorithm Key Type Strengths Use Cases
HS256/HS384/HS512 Symmetric (shared secret) Fast; simple for internal systems. Stateless sessions in single-server apps.
RS256/RS384/RS512 Asymmetric (RSA public/private) Secure distribution; non-repudiation. API tokens (e.g., OAuth ID tokens).
ES256/ES384/ES512 Asymmetric (ECC) Smaller keys; mobile-efficient. IoT or high-volume signing.
PS256 Asymmetric (RSA-PSS) Probabilistic padding; quantum-resistant potential. High-security environments.

Use Cases

  • Software Updates: Sign binaries (e.g., Microsoft signs .NET packages) to prevent malware.
  • Email Signing: PGP/S/MIME signs messages for trust.
  • Blockchain Transactions: ECDSA signs transfers (e.g., Bitcoin) for ownership proof.
  • JWT in Auth: Signed tokens for offline verification in distributed systems (e.g., microservices).

Digital signatures are tamper-proof "seals"—essential for trust in digital comms. Always use secure keys and rotate them!
