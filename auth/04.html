Access Control Models: Overview and Differences

Access control models define how permissions are granted to users, systems, or processes to access resources. They balance security, flexibility, and manageability. Below, I'll list four common models—RBAC (Role-Based Access Control), ABAC (Attribute-Based Access Control), ACL (Access Control Lists), and ReBAC (Relationship-Based Access Control)—with brief descriptions. Then, a comparison table highlights their key differences.

1. RBAC (Role-Based Access Control)

  • Description: Permissions are assigned to roles (e.g., "Admin", "User"), and users are assigned to roles. Access is granted based on the user's role. It's hierarchical (roles can inherit from others) and widely used in enterprise systems.
  • Example: A "Manager" role allows viewing reports; users in that role inherit the permission.

2. ABAC (Attribute-Based Access Control)

  • Description: Decisions use attributes (e.g., user location, time, resource sensitivity) evaluated against policies. A policy engine (e.g., XACML) dynamically computes access.
  • Example: Allow access if user.department = "HR" AND time > 9 AM AND resource.sensitivity = "low".

3. ACL (Access Control Lists)

  • Description: Resources have explicit lists of users/groups and their permissions (e.g., read/write). It's resource-centric and simple for small-scale setups.
  • Example: A file's ACL lists "user1: read, write" and "group2: read-only".

4. ReBAC (Relationship-Based Access Control)

  • Description: Access is based on relationships between entities (e.g., user owns resource, or user is a follower of owner). Popular in graph-based systems like Google's Zanzibar.
  • Example: Allow "editor" access if user is "member_of" the same "team" as the document owner.

Key Differences

Model Granularity Flexibility Complexity Best For Drawbacks
RBAC Role-level (coarse) Medium (static roles) Low (easy to audit) Enterprises with stable hierarchies (e.g., HR systems) Rigid; doesn't handle dynamic contexts well
ABAC Attribute-level (fine) High (dynamic policies) High (policy engine needed) Complex, context-aware apps (e.g., cloud IAM) Performance overhead; hard to debug policies
ACL User/resource-level (fine) Low (explicit lists) Low (simple lists) Small-scale filesystems or apps (e.g., local servers) Scalability issues; hard to manage at scale
ReBAC Relationship-level (fine) High (graph-based) Medium-High (needs graph DB) Social/collaborative systems (e.g., Google Drive sharing) Requires modeling relationships; query complexity

These models can be combined (e.g., RBAC + ABAC for hybrid control). Choose based on your system's scale, dynamics, and compliance needs—RBAC for simplicity, ABAC for nuance. If implementing in .NET, libraries like IdentityServer or Auth0 support these.
