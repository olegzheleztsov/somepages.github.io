What is Infrastructure as Code (IaC)?

Infrastructure as Code (IaC) is the practice of managing and provisioning computing infrastructure through machine-readable definition files (e.g., code) rather than manual UI clicks or scripts. It treats infrastructure (servers, networks, databases) like application code: version-controlled, tested, reviewed, and deployed via CI/CD pipelines.

In .NET ecosystems:

  • Tools like Terraform, ARM templates (for Azure), or Pulumi automate spinning up resources for .NET apps (e.g., provisioning an AKS cluster for microservices).
  • It's declarative (define desired state) or imperative (step-by-step commands), enabling reproducibility across envs.

Think of it as Git for your cloud: Commit a change, and your infra updates safely—no more "sudo" roulette.

Why IaC? (Quick Value Prop)

  • Consistency: Eliminates "works on my machine" for infra—dev/staging/prod match exactly.
  • Speed & Safety: Automate deploys with rollbacks; audit trails via Git.
  • Scalability: As a Tech Lead, it empowers self-service for teams, reducing bottlenecks.
  • Ties to config mgmt (e.g., Ansible) by composing infra + app configs.

Core IaC Concepts

Here's a breakdown of foundational concepts, with .NET-flavored examples:

Concept Description .NET Example
Declarative vs. Imperative Declarative: Specify end-state (tool figures "how"); Imperative: Script exact steps. Declarative is safer for drift detection. Declarative: Terraform HCL file for Azure SQL DB. Imperative: PowerShell script to create it.
Idempotency Running the code multiple times yields the same result—no side effects. A Terraform apply on an existing .NET App Service won't recreate it; just verifies.
State Management Tracks current infra state (e.g., in a file/backend) to compute changes. Terraform's terraform.tfstate file (store in Azure Blob for teams) diffs against your code.
Version Control Infra code in Git; branches for envs, PRs for reviews. Branch feature/new-api adds a .NET Function App; merge triggers deploy.
Modularity & Reusability Break into modules/roles (e.g., reusable VPC or DB templates). Terraform modules for a shared .NET Redis cache across services.
Immutable Infrastructure Treat resources as disposable—rebuild vs. mutate for reliability. Bake .NET app into Docker images; spin up new EC2 instances on deploy, terminate old.
Policy as Code Embed compliance checks (e.g., no public S3 buckets) into pipelines. OPA (Open Policy Agent) gates a .NET pipeline if ARM template violates tagging rules.
Orchestration vs. Provisioning Provisioning: Create resources (e.g., VMs). Orchestration: Coordinate (e.g., with Ansible post-provision). Pulumi provisions Azure resources, then Ansible configures IIS for .NET.

Pro Tip for Your Interview: Emphasize integration—"In a .NET pipeline, I'd use Azure Bicep (declarative ARM) for IaC, validated with Pester tests, to provision scalable App Services." Scenario: "Migrating a monolith to IaC cut our infra drift incidents by 80%." Highlights leadership in DevOps maturity.
