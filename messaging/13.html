What is a Topic in Kafka?

A topic in Kafka is a logical abstraction—a named stream or category where producers publish records (messages) and consumers subscribe to read them. It acts as the primary unit of organization in Kafka, similar to a "channel" or "feed" in messaging systems. Topics are append-only, immutable logs that store records in the order they arrive, enabling durable, ordered event storage.

  • Key Characteristics:
    • Naming: Topics have user-defined names (e.g., "user-events" or "orders").
    • Durability: Configurable retention policies (e.g., time-based like 7 days, or size-based like 1GB) determine how long records are kept.
    • Pub/Sub Semantics: Multiple producers can write to a topic, and multiple consumer groups can read from it independently—supporting fan-out without interference.
    • Configuration: When creating a topic, you specify replication factor (for fault tolerance) and number of partitions (for scalability).

Topics are the entry point for data in Kafka ecosystems, used for everything from real-time logs to event sourcing in microservices.

What is a Partition in Kafka?

A partition is the fundamental unit of scalability and parallelism within a Kafka topic. Each topic is divided into one or more partitions, where each partition is an ordered, immutable sequence of records (a commit log) stored on disk across brokers. Partitions allow Kafka to distribute load horizontally: Records are assigned to partitions based on a key (for ordering) or round-robin (for even distribution).

  • Key Characteristics:
    • Ordering: Strict FIFO order within a partition—records can't be rearranged or deleted (only compacted or truncated).
    • Distribution: Partitions are spread across brokers in the cluster; one partition leader handles reads/writes, with replicas for redundancy.
    • Offsets: Each record gets a unique offset (position) per partition, allowing consumers to track progress and replay from any point.
    • Scalability: More partitions = more parallelism (e.g., multiple consumers can process different partitions simultaneously).

Partitions make Kafka handle massive throughput (e.g., millions of messages/sec) by enabling horizontal scaling without global ordering trade-offs.
